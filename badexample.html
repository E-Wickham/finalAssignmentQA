Bad Examples

The previous form on this page is a basic web form that is a security/QA nightmare for several reasons.

  1. The password input is not hidden. You see the characters being typed into the input box.
  2. The form is a GET method form, which appends the form info to the address bar in a very obvious way.
  3. There is no form validation on either the client side or the server side.
  4. No password restrictions are being used to create more secure logins
  5. If this was connected to a database you best believe the inputs would not be sanitized and they would get blown up immediately

Solutions

Some very simple solutions to this are: Hide the password input, require client and server side validation.

Require upper and lowercase letters, one number and one special character or another proven set of characters that increase the password strength.

Hashing passwords in the db will help this. Switch the form to post, collect it somewhere and sanitize the input so the the eventual db won't be vulnerable to sql injection.
